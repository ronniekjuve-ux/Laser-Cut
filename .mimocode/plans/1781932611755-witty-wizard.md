# Plan: Слияние раскладок (Layout Merge)

## Context

Оператор хочет объединять раскладки из разных заказов, когда детали можно разместить на одном листе. Пример: раскладки А2 и Б2 содержат детали, которые можно соместить → загружается новая раскладка Н1 с этими деталями. Старые раскладки помечаются как заменённые, создаётся новый объединённый заказ.

## Дизайн-решения

- Новый заказ создаётся как отдельный Application (имя: "Слияние: ЗаказА + ЗаказБ")
- Статус объединённого заказа: `pending` (как обычная заявка)
- При пересорте — предупреждение с таблицей расхождений, но можно продолжить
- Заменённые раскладки отображаются серым/блёклым, без кнопок статуса
- В аудите replaced-раскладки исключаются из агрегации

## Изменения

### 1. Модель ApplicationLayout — два новых поля

**Файл:** `app/db/models.py`

```python
class ApplicationLayout(Base):
    # ... существующие поля ...
    status: Mapped[str] = mapped_column(String(20), default="active")       # "active" | "replaced"
    merged_from: Mapped[Optional[str]] = mapped_column(Text, nullable=True)  # JSON: {apps: [...], layouts: [...]}
```

**Миграция:** `ALTER TABLE application_layouts ADD COLUMN status VARCHAR(20) DEFAULT 'active'; ALTER TABLE application_layouts ADD COLUMN merged_from TEXT;`

### 2. Бэкенд: Эндпоинт слияния

**Файл:** `app/api/v1/router_applications.py`

Новый эндпоинт `POST /api/v1/applications/merge`:

```
Вход (multipart/form-data):
  - file: .doc раскладка (новая)
  - layout_ids: JSON массив [{app_id, layout_id}, ...] — какие раскладки заменяются

Логика:
1. Сохранить загруженный .doc файл
2. Распарсить новую раскладку (parse_layout_text)
3. Собрать parts из всех исходных раскладок (из БД)
4. Сверить: каждый part из новой раскладки должен быть в одном из исходных
   - Совпадение по normalize_name(name) + dx + dy
   - quantity новой >= суммарного quantity исходных
5. Если расхождения — вернуть предупреждения (warnings), но не блокировать
6. Определить набор уникальных customer_id из исходных заявок
7. Создать новый Application:
   - order_name: "Слияние: {имя1} + {имя2}"
   - customer_id: первый уникальный (остальные — в comments)
   - material/thickness: из новой раскладки
   - status: "pending"
   - supply_material: True если хотя бы один исходный — давальский
8. Создать ApplicationLayout с:
   - status: "active"
   - merged_from: JSON {apps: [...], layouts: [...]}
   - parsed layout data + parts
9. Пометить исходные раскладки: status = "replaced"
10. Вернуть: {new_app_id, warnings: [...]}
```

### 3. Бэкенд: Исключение replaced из списка

**Файл:** `app/api/v1/router_applications.py`

В `list_applications` (строка ~380): добавить фильтр `ApplicationLayout.status == "active"` при определении machine type.

В `get_application_details` (строка ~760): layouts с `status == "replaced"` возвращаются, но с флагом `replaced: true`. Frontend отображает их серым.

В аудите `router_audit.py`: добавить `WHERE status = 'active' OR status IS NULL` при загрузке раскладок.

### 5. Фронтенд: Модалка слияния

**Файл:** `frontend/src/pages/Applications/MergeModal.jsx` (новый)

Этапы:
1. **Выбор раскладок**: Таблица со всеми активными раскладками из обоих вкладок (Заявки + Заказы). Чекбоксы для выбора 2+ раскладок. Отображается: имя заказа, код раскладки, станок, список деталей.
2. **Загрузка новой раскладки**: File input для .doc файла.
3. **Проверка**: После загрузки — запрос на `POST /merge`. Отображение warnings если есть расхождения.
4. **Подтверждение**: Кнопка "Слияние" → вызов эндпоинта.

### 6. Фронтенд: Отображение replaced-раскладок

**Файл:** `frontend/src/pages/Applications/ApplicationDetail.jsx`

В списке раскладок: раскладки с `replaced: true` отображаются с:
- `opacity: 0.4`
- Серый фон
- Нет кнопок статуса/приоритета
- Бейдж "Заменена"
- Стрелка → ссылка на объединённый заказ (new_app_id из merged_from)

### 7. Аудит: Фильтрация replaced

**Файл:** `app/api/v1/router_audit.py`

В запросе раскладок (строка ~177): добавить `ApplicationLayout.status.in_(["active", None])` — исключить replaced.
