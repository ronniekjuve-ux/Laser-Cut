# Исправление парсинга деталей, веса и изображений

## Найденные проблемы

### 1. Парсинг деталей из заявки не работает (`parts_count: 0`)
Причина: формат LibreOffice таблицы в raw text:
```
|      |Имя детали                                     |DX       |DY       |Кол-во  |
|3     |D:\...\8ММ-1-657-4ШТ      |120      |906      |4       |
|      |ПЛОЩ РАСКОСЫ В ОСЯХ 8-9\8ММ\8ММ-1-657-4ШТ      |         |         |        |
```
- Каждая деталь занимает **2 строки**: первая = № + путь + DX + DY + Qty, вторая = продолжение пути
- Парсер в `parse_application_text()` ищет `len(cells) >= 7` (строка 385), но формат с `|` разделителем даёт меньше ячеек
- Парсер в `parse_layout_text()` обрабатывает формат с `|` (строки 518-631), но пытается найти путь в `path_cell` — при двустрочном формате путь обрезается

### 2. Вес деталей не отображается
- При первой загрузке `app.thickness = 0.0` → вес = None
- После загрузки раскладки толщина обновляется (строка 248-254), но веса деталей уже записаны как None
- **Решение:** пересчитать веса после обновления толщины

### 3. Изображения перепутаны
- `extract_images()` извлекает изображения из .doc через LibreOffice → `image1.png`, `image2.png` и т.д.
- Порядок изображений совпадает с порядком деталей в таблице заявки
- Но изображения привязываются к деталям раскладки по индексу, а порядок деталей в раскладке может отличаться
- **Решение:** сопоставлять по `normalize_name()` — имя из пути раскладки должно совпадать с именем из заявки

## Изменения

### 1. `app/services/unified_parser.py` — исправить парсинг двухстрочного формата

#### В `parse_application_text()` (строки 361-438):
Заменить блок парсинга деталей. Текущий код пытается ловить `|`-таблицу с `len(cells) >= 7`, но формат LibreOffice даёт строки вида:
```
|3     |D:\...\8ММ-1-657-4ШТ      |120      |906      |4       |
```
Нужно: извлекать путь из `cells[1]`, извлекать DX/DY/Qty из остальных ячеек, обрабатывать двухстрочный формат (вторая строка = продолжение пути).

Новый алгоритм:
```python
# Ищем строки с путями (содержат D:\ или C:\)
# Каждая такая строка — начало новой детали
# Извлекаем: имя из пути, DX/DY/Qty из ячеек этой же строки
# Если вторая строка тоже содержит путь — это продолжение (пропускаем)
```

#### В `parse_layout_text()` (строки 518-631):
Аналогично — текущий код обрабатывает `|`-формат, но при двустрочном формате:
- Строка 1: `|3     |D:\...\8ММ-1-657-4ШТ      |120      |906      |4       |`
- Строка 2: `|      |ПЛОЩ РАСКОСЫ В ОСЯХ 8-9\8ММ\8ММ-1-657-4ШТ      |         |         |        |`

Текущий код (строка 547) ищет `path_cell` и пытается извлечь числа. При двустрочном формате:
- Строка 1: путь найден, `cells[1]` = `D:\...\8ММ-1-657-4ШТ      `, числа = `[120, 906, 4]` → работает!
- Строка 2: путь НЕ найден (нет `A:\`), `cells[1]` = `ПЛОЩ РАСКОСЫ...` → текущий код пропускает (нет `path_cell`)

**Проблема:** в текущем коде (строка 551) `if path_cell:` блок обрабатывает только строки с путями. Строка 2 (продолжение) не обрабатывается — но числа DX/DY/Qty уже есть в строке 1!

**Решение:** код уже должен работать для формата `|3|D:\path|120|906|4|`, потому что:
- `path_cell` = `D:\...\8ММ-1-657-4ШТ`
- `numbers` = `[120, 906, 4]` (из ячеек после пути)
- Деталь создаётся корректно

Но в `parse_application_text()` формат другой — нет `|`-разделителя в том же виде. Нужно адаптировать.

### 2. `app/api/v1/router_applications.py` — пересчёт весов

После строки 254 (после второго `db.commit()`), добавить:
```python
# Пересчёт весов с учётом обновлённой толщины
if app.thickness and app.thickness > 0:
    parts_res = await db.execute(
        select(ApplicationLayoutPart).where(ApplicationLayoutPart.layout_id == layout.id)
    )
    for part in parts_res.scalars().all():
        if part.dx > 0 and part.dy > 0:
            part.weight = round(part.dx * part.dy * app.thickness * 7.85 / 1_000_000, 4)
    await db.commit()
```
