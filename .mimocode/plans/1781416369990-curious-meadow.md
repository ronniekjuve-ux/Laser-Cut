# RBAC и роль-ориентированный интерфейс

## Архитектурный обзор

Текущее состояние: роли есть в `User.role`, но авторизация НЕ настроена на大部分 endpoints, `Customer` не связан с `User`, статусов у `Application` нет.

## Фаза 1: Модели данных

### 1.1 Связь User ↔ Customer
**Файл:** `app/db/models.py`
- Добавить `customer_id: Mapped[Optional[int]] = mapped_column(ForeignKey("customers.id"), nullable=True)` в `User`
- Добавить relationship `customer: Mapped[Optional["Customer"]]`

**Миграция:** `ALTER TABLE users ADD COLUMN customer_id INTEGER REFERENCES customers(id);`

### 1.2 Статус и приоритет заявки
**Файл:** `app/db/models.py`
- Добавить `ApplicationStatus` enum: `pending`, `in_progress`, `partially_cut`, `cut`
- Добавить `ApplicationPriority` enum: `low`, `medium`, `high`, `urgent`
- Добавить `status`, `priority`, `updated_by` в `Application`

**Миграция:**
```sql
CREATE TYPE applicationstatus AS ENUM ('pending','in_progress','partially_cut','cut');
CREATE TYPE applicationpriority AS ENUM ('low','medium','high','urgent');
ALTER TABLE applications ADD COLUMN status applicationstatus DEFAULT 'pending';
ALTER TABLE applications ADD COLUMN priority applicationpriority DEFAULT 'medium';
ALTER TABLE applications ADD COLUMN updated_by INTEGER REFERENCES users(id);
```

### 1.3 Дефицит металла
**Файл:** `app/db/models.py` — новая модель `DeficitRequest`
```python
class DeficitRequest(Base):
    id, application_id (FK), material, thickness, size, quantity,
    customer_name, status (pending/resolved), note,
    created_by (FK users), created_at
```

### 1.4 Уведомления
**Файл:** `app/db/models.py` — новая модель `Notification`
```python
class Notification(Base):
    id, user_id (FK), type (status_change/deficit),
    message, is_read, related_app_id, created_at
```

## Фаза 2: Backend — Авторизация

### 2.1 Защита endpoints
**Файл:** `app/api/v1/router_applications.py`
- `POST /upload` → `require_role(ADMIN)`
- `POST /{id}/layouts/upload` → `require_role(ADMIN)`
- `GET /` → `get_current_user` + фильтрация по роли:
  - admin/director: все заявки
  - operator: все заявки
  - customer: только свои (по `customer_id`)
- `GET /{id}` → `get_current_user` + проверка доступа
- `DELETE /{id}` → `require_role(ADMIN)`
- **НОВЫЙ** `PATCH /{id}/status` → `require_role(ADMIN, OPERATOR)` — изменение статуса

## Фаза 3: Frontend — Роль-ориентированный UI

### 3.1 Навигация по ролям
**Файл:** `frontend/src/components/Layout.jsx`
```jsx
const NAV_ITEMS = [
  { to: '/', label: 'Заявки', icon: '📋', roles: ['admin','director','operator','customer'] },
  { to: '/deficit', label: 'Дефицит', icon: '⚠️', roles: ['admin','director','operator','customer'] },
  { to: '/schedule', label: 'График', icon: '📅', roles: ['admin','director','operator'] },
  { to: '/users', label: 'Пользователи', icon: '👥', roles: ['admin','director'] },
  { to: '/audit', label: 'Аудит', icon: '🔐', roles: ['admin'] },

```

### 3.2 Фильтрация данных по роли
**Файл:** `frontend/src/pages/Applications/ApplicationsList.jsx`
- Заказчик видит только свои заявки (backend фильтрует)
- Оператор может менять статус (кнопки рядом с заявкой)
- Директор может ставить приоритет

### 3.3 UI смены статуса (оператор)
В `ApplicationDetail.jsx` — добавить кнопки статуса:
- `В очереди` → `В работе` → `Вырезано частично` → `Вырезано`
- При смене — сохраняется `updated_by`

### 3.4 UI приоритетов (директор)
В `ApplicationsList.jsx` — столбец "Приоритет" с текстовыми метками:
- `low` → серый бейдж "Низкий"
- `medium` → синий бейдж "Средний"  
- `high` → оранжевый бейдж "Высокий"
- `urgent` → красный бейдж "Срочно"
- Клик по бейджу → циклически меняет приоритет (для director/admin)

### 3.5 Заявка дефицита
Кнопка "Заказать материал" в `ApplicationDetail` или `Deficit` странице. Форма: материал, толщина, размер, количество.

### 3.6 Уведомления
В `Layout.jsx` — колокольчик с счётчиком непрочитанных. Клик → dropdown со списком уведомлений.

## Верификация
1. Логин как admin → видит всё, может всё
2. Логин как operator → видит заявки/дефицит/график, может менять статус
3. Логин как customer → видит только свои заявки
4. Логин как director → видит всё, ставит приоритеты
5. Смена статуса оператором → уведомление заказчику
6. Создание дефицита → отображается в списке
7. `POST /users/` без токена → 403
8. `GET /applications/` как customer → только свои
